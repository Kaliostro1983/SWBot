FAQ / Довідка

Відповіді на часті питання по роботі WS Bridge.

📱 WhatsApp

Причина. WhatsApp Web.js керує браузером Chrome через Puppeteer. Послідовність старту виглядає так:

Запуск Chrome → Завантаження WhatsApp Web → Authenticated → [завантаження JS/DOM] → Client ready

Якщо сесія збережена (папка .wwebjs_auth), крок «Authenticated» проходить одразу. Але далі браузер може зависнути на етапі завантаження сторінки — WhatsApp Web не відповідає подією ready. Про це свідчить lastReadyAt: null у стані.

Найчастіша причина — WhatsApp оновив свій веб-додаток, і бібліотека тимчасово не може завершити ін'єкцію свого JS у сторінку. Також може бути: повільний інтернет, Chrome не встиг запуститися в headless-режимі.

Кроки усунення (по порядку):

  • 1 Перезапустити клієнт — панель → Інтеграція → карта WhatsApp → кнопка ↺ (або через API). Зачекати 20–30 сек. Вирішує у 90% випадків.
  • 2 Видалити стару сесію — зупинити сервіс, видалити папку .wwebjs_auth/, запустити знову і відсканувати QR-код. Допомагає якщо сесія застаріла або пошкоджена.
  • 3 Оновити бібліотеку — якщо проблема повторюється регулярно, вийшов патч:
npm install whatsapp-web.js@latest
💡 Як розпізнати цю проблему у логах: є подія Authenticated, але немає Client ready протягом 3–5 хвилин після старту.
⚠️ Не копіюйте папку .wwebjs_auth/ між комп'ютерами — сесія прив'язана до машини. На новому ПК завжди проходьте QR заново.

WhatsApp Web розриває з'єднання якщо телефон довго офлайн, або WhatsApp розлогінив пристрій вручну. Сервіс автоматично намагається перепідключитись.

Якщо автоматичне перепідключення не допомогло — перезапустіть клієнт через кнопку ↺ в панелі. Якщо з'явиться QR-код — відскануйте його знову.

💡 Переконайтесь що телефон з WhatsApp онлайн і не вимкнено інтернет. Пов'язані пристрої можуть автономно працювати до 14 днів без телефону.

Можливі причини:

  • 1Chrome не встановлений або встановлений у нестандартну папку. Перевірте логи — там буде шлях до браузера.
  • 2Антивірус блокує Puppeteer. Додайте папку проєкту у виключення.
  • 3Стара сесія ще активна — клієнт авторизується без QR. Якщо хочете примусово отримати QR — видаліть .wwebjs_auth/.

Для діагностики тимчасово увімкніть видимий Chrome — в .env поставте HEADLESS=0 і перезапустіть сервіс.

Симптом. Після запуску сервісу WhatsApp показує «connecting» / «Підключення…» протягом 5+ хвилин. У логах є рядок Bot initialize requested, але далі — повна тиша. Немає ні Authenticated, ні Client ready, ні жодних помилок.

🔴 Причина — застаріла або пошкоджена сесія. Папка .wwebjs_auth/ містить дані старої сесії. Chrome запускається, намагається відновити сесію, але WhatsApp Web її відхиляє, не надсилаючи жодних подій назад у бібліотеку. Бот «завис» в очікуванні події, яка ніколи не прийде.

Як це виправити:

  • 1 Зупиніть сервіс — закрийте термінал або натисніть Ctrl+C.
  • 2 Видаліть папку сесії у корені проєкту:
# Windows (PowerShell)
Remove-Item -Recurse -Force .wwebjs_auth, .wwebjs_cache

# або через Провідник файлів — видаліть папки .wwebjs_auth та .wwebjs_cache
  • 3 Запустіть сервіс знову — node index.cjs.
  • 4 Відскануйте QR-код у панелі → Інтеграція → WhatsApp. Нова чиста сесія буде збережена.
💡 Як розпізнати цю проблему: у логах є Bot initialize requested, але після нього — тиша більше 2 хвилин. Немає навіть рядка Authenticated. Це відрізняється від звичайного повільного старту, де Authenticated з'являється, але Client ready затримується.
⚠️ Щоб уникнути повторення — не переміщуйте .wwebjs_auth/ між машинами і не відновлюйте із резервних копій. Сесія прив'язана до конкретного пристрою та часу реєстрації.
🔵 Signal

Signal-міст працює через два Docker-контейнери. Якщо Docker Desktop не запущений або контейнери зупинені — Signal не може підключитись.

  • 1Відкрийте Docker Desktop і переконайтесь що він у статусі «Running».
  • 2Запустіть контейнери:
docker compose -f docker-compose.signal.yml up -d
  • 3Зачекайте 10–15 секунд — Signal у панелі покаже connected.

Так, це нормальна поведінка. Signal-міст працює через Docker-контейнер signal-cli-api, який побудований на Java (JVM). При першому старті JVM виконує JIT-компіляцію і споживає до 600–650% CPU протягом 1,5–3 хвилин. Поки цей процес не завершиться — контейнер не відповідає на запити, і бот бачить Signal як «connecting».

💡 Бот автоматично повторює спроби підключення кожні 5 секунд. Після того як JVM прогріється — Signal підключиться сам, без будь-яких дій з вашого боку.

Як прискорити та не повторювати щоразу:

  • 1 Увімкніть автозапуск Docker Desktop при вході у Windows: Docker Desktop → Settings → General → ✅ Start Docker Desktop when you sign in to your computer.
  • 2 Контейнери запустяться автоматично — у docker-compose.signal.yml вже стоїть restart: unless-stopped, тому після перезапуску Windows Docker сам підніме signal-cli-api і signal-bridge.
  • 3 JVM прогріється до того, як ви відкриєте панель — якщо комп'ютер стартував кілька хвилин тому, Signal вже буде готовий.
✅ Після першого «прогріву» JVM кешує скомпільований код. Якщо контейнер не зупиняти (тільки бот перезапускати) — підключення до Signal займає 3–5 секунд.

Затримка залежить від двох параметрів у .env та docker-compose.signal.yml:

# .env — як часто Node.js опитує Signal bridge
SIGNAL_POLL_MS=1000

# docker-compose.signal.yml — таймаут long-poll до signal-cli
SIGNAL_RECEIVE_TIMEOUT_SEC=20

Оптимальні значення: SIGNAL_POLL_MS=1000 + SIGNAL_RECEIVE_TIMEOUT_SEC=20. Після зміни SIGNAL_RECEIVE_TIMEOUT_SEC потрібно перебудувати контейнер:

docker compose -f docker-compose.signal.yml up -d --build signal-bridge

Дані Signal зберігаються у Docker volume signal_cli_data. Якщо ви не перенесли volume — так, потрібна повторна прив'язка через QR у панелі → Інтеграція → Signal → «Прив'язати».

Якщо хочете зберегти прив'язку при перенесенні — експортуйте volume:

# На старому ПК — зберегти
docker run --rm -v signal_cli_data:/data -v %cd%:/backup alpine tar czf /backup/signal_data.tar.gz /data

# На новому ПК — відновити
docker run --rm -v signal_cli_data:/data -v %cd%:/backup alpine tar xzf /backup/signal_data.tar.gz -C /
⚡ Автоматизації

Перевірте послідовно:

  • 1Обидва месенджери підключені — статус connected у вкладці Інтеграція.
  • 2Автоматизація не на паузі — кнопка паузи у картці flow.
  • 3Правильний чат-джерело — перевірте налаштування автоматизації, чи обраний потрібний чат.
  • 4Фільтр ключових слів — якщо задані ключові слова, повідомлення має їх містити. Для пропуску всіх — поставте *.
  • 5Логи — вкладка «Логи» в панелі покаже де саме зупиняється обробка.

У налаштуваннях автоматизації є окремий чекбокс для медіа. Перевірте що він увімкнений:

Панель → Автоматизації → редагувати flow → вкладка «Додатково» → ✅ «Пересилати зображення з повідомлення».

💡 Якщо ви відправляєте фото і текст окремими повідомленнями (не як фото з підписом) — бот обробляє їх як два окремих повідомлення. Текстове повідомлення не містить медіа (hasMedia: false), тому пересилається без фото. Надсилайте фото разом із підписом — тоді всі дані в одному повідомленні.

Так. Сервіс підтримує кілька автоматизацій з одним і тим самим чатом-джерелом — усі вони спрацьовують одночасно при отриманні повідомлення.

Наприклад: один чат Signal може одночасно пересилати у групу WhatsApp A і у групу WhatsApp B через дві окремі автоматизації.

✅ Просто створіть ще одну автоматизацію з тим самим чатом-джерелом і іншим цільовим чатом.
⚙️ Загальне

Інший процес вже використовує порт 3001 (наприклад, запущена ще одна копія сервісу).

# Знайти процес на порту 3001
netstat -ano | findstr :3001

# Завершити процес за PID
taskkill /PID <номер> /F

Або змініть порт у .env: PORT=3002 (але тоді і Signal bridge потрібно перенести на інший порт).

Всі дані зберігаються у папці data/ у корені проєкту:

data/flows.json          ← автоматизації
data/chat-directory.json  ← каталог чатів
data/whatsapp-chats-cache.json ← кеш чатів WA
data/signal-chats-cache.json  ← кеш чатів Signal

При перенесенні на інший ПК скопіюйте цю папку щоб зберегти всі налаштування.

Три місця де шукати інформацію:

  • 1Панель → вкладка «Логи» — всі події в реальному часі з фільтрацією по рівню (INFO / WARN / ERROR).
  • 2Термінал де запущений node index.cjs — повний stdout із додатковими [ROUTING] та [ROUTER CHECK] мітками.
  • 3Docker Desktop → контейнер signal-bridge → вкладка Logs — логи Signal-моста.
💡 Файл logs/signal_raw.ndjson містить сирі Signal-повідомлення у форматі NDJSON — корисно для перевірки що саме приходить від signal-cli.
