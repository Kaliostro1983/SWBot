## Перед `POST` на FastAPI

Повідомлення з чату спочатку перевіряються на **фільтри автоматизації** (поля «ключові слова» та «частоти» як підрядки в повному тексті повідомлення WhatsApp). Якщо заповнені **обидва** поля — достатньо збігу **або** по ключових словах, **або** по частотах; якщо лише одне — діє воно. Якщо умови не виконані — виклик API **не відбувається**. Токен `*` у будь-якому з цих полів вимикає фільтрацію для цього запису. У режимі лише `.env` можна задати `SOURCE_FILTER_KEYWORDS` / `SOURCE_FILTER_FREQUENCIES`.

## Що надсилає бот

| Поле | У бекенді | Примітка для WA Bridge |
|------|-----------|-------------------------|
| `chat_id` | Обов’язкове | `msg` → id чату-джерела |
| `message_id` | Обов’язкове | `msg.id._serialized` |
| `text` | Обов’язкове, **не порожній** | Текст **після** зняття префікса `SEND_PREFIX` (`#go`). Порожній текст на бекенд **не відправляємо** (див. код). |
| `platform` | Ні, за замовчуванням `whatsapp` | Передаємо `whatsapp` |
| `chat_name` | Ні | Зараз `null` (можна розширити) |
| `published_at_platform` | Ні | ISO з timestamp повідомлення |
| `allow_send` | Ні, за замовчуванням `false` | `true`, якщо вихідний текст **починається** з `SEND_PREFIX` |
| `flow_id`, `flow_name` | **Не читаються** сервером | Додаються для логів/маршрутизації на боці бота; FastAPI ігнорує |

Додатково в тіло може потрапляти `author` — у довіднику бекенду не обов’язкове; зайві ключі не ламають обробку.

## Відповідь

- **HTTP 400** при невалідному мінімумі: тіло на кшталт `{ "ok": false, "error": "…" }`, **`actions` немає**.
- Успіх: у відповіді зазвичай є **`actions`** (масив).
- Зараз сервер додає лише  
  `{ "type": "send_message", "text": "<той самий text з запиту>" }`  
  якщо **`allow_send === true`** і **`text`** непорожній; інакше **`actions`** часто **`[]`**.

Бот обходить `actions` і для `type === "send_message"` відправляє `text` у чат з **`TARGET_CHAT`** у `.env` (або з автоматизації, коли це знову буде підтримано в моделі).

## Навіщо `SEND_PREFIX` (`#go`) і `allow_send`

Під час одного невдалого запуску чат-бот почав розсилати повідомлення не в цільовий чат, а широко. Тому на етапі тестування додано **два важелі**:

1. **`allow_send`** — передається в RER; сервер **не додає** `send_message` у `actions`, якщо `allow_send` false.
2. **Префікс `#go`** на боці бота — лише якщо повідомлення **явно** позначене префіксом, `allow_send` стає `true` (підлаштування під ваш сценарій тестування).

У проді правила можна змінити (наприклад інший префікс або завжди `true` після перевірок).
