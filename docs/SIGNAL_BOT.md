# SIGNAL BOT

## Функції Signal-бота

- Інтеграція Signal через `signal-cli-rest-api` + `signal-bridge`.
- Перевірка стану інтеграції: `health`, `linked`, робота polling-процесу.
- Отримання списків чатів Signal для UI (з фільтрацією технічних ID за умовами).
- Отримання вхідних повідомлень Signal через polling.
- Маршрутизація повідомлень за automation-правилами:
  - `Signal -> WhatsApp`
  - `Signal -> FastAPI`
  - `Signal -> Signal`
- Відправка повідомлень у Signal (`/send`) та в WhatsApp (через WA-клієнт).
- Підтримка лінкування через QR (`/link`) та ручного перелінкування.

## Що вже зроблено

- Стабілізовано Signal onboarding і процес генерації QR через bridge-first підхід.
- Додано перевірку `linked` та окремий контрольний endpoint для UI.
- Додано захист від паралельних link/poll запитів і throttling шумних логів.
- Підвищено таймаути для повільних інстансів Signal API.
- Додано автостарт сервісу з безпечним режимом Signal (без автоперелінкування за замовчуванням).
- Перероблено UI вибору чатів у модалці automation (search + chips).
- Додано збереження пар `chatId + name` у flow-конфігурації.
- Додано персистентний `chat-directory` для назв чатів.
- Додано dual-id/aliases підхід для Signal source refs у flow.
- Додано список нещодавно активних Signal-чатів у state для швидкого вибору джерела в UI.

## Що треба зробити

- Дотиснути стабільний deterministic matching для Signal source після relink/перезапусків.
- Додати явний UI-індикатор mismatch: "повідомлення отримуються, але не з чатів-джерел automation".
- Додати простий UX для швидкого прив'язування джерела без ручного аналізу логів:
  - кнопка/чіп "обрати останній активний чат як source".
- Зафіксувати E2E smoke-тести для:
  - `Signal -> WhatsApp`
  - `Signal -> Signal`
  - `Signal -> FastAPI`
- Додати короткий runbook "що робити при зміні Signal-акаунта" (без автоматичного ризикового remap).

## Проблемний поточний таск

Проблема виникла на задачі:
"Пересилати повідомлення тільки з конкретно вибраного Signal-чату в конкретний WhatsApp-чат без ручного пошуку `chatId` після перезапусків/relink".

Симптоми:

- `Signal messages fetched` є, тобто transport працює.
- `Minute activity.received` залишається `0`.
- Логи містять `Signal message ignored: no matching source chat`.
- Вхідні `chatId` часто не збігаються з `sourceChatIds` у flow (різні формати ID/оновлення ID після relink).

Поточний статус:

- Авто-ремап джерел за замовчуванням вимкнено (`AUTO_SIGNAL_SOURCE_AUTOREMAP=0`) для безпечного strict-режиму.
- Потрібно завершити UX-частину, щоб користувач міг вибирати правильне джерело "в 1-2 кліки" без ручного розбору логів.
